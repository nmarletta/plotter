# Story 1.1: Implement SerialManager for GRBL Serial Communication

## Status
In Progress

## Story
**As a** plotter controller developer,
**I want** a SerialManager component that handles bidirectional serial communication with GRBL,
**so that** all other components (GCodeStreamer, StateSettings, StateControl) have a reliable, tested foundation for GRBL communication.

## Acceptance Criteria

1. SerialManager initializes Serial1 at 115200 baud and verifies GRBL connection readiness
2. SerialManager can send G-code commands to GRBL and wait for acknowledgment (ok/error)
3. SerialManager implements timeout-based flow control (1 second default timeout)
4. SerialManager parses GRBL responses and distinguishes between ok, error, and alarm messages
5. SerialManager provides error message retrieval for display to user
6. SerialManager can query GRBL readiness status without blocking
7. All serial operations are non-blocking and return within timeout period
8. Existing functionality (file browser, display, encoder) remains operational after integration

## Dev Notes

### Architecture Context
[Source: docs/architecture.md#Component 1: SerialManager]

SerialManager is the **foundation component** for all GRBL communication in this brownfield enhancement. It abstracts the Serial1 hardware UART and implements the GRBL 1.1f protocol flow control. All other components (GCodeStreamer, StateSettings, StateControl) will depend on SerialManager.

**Critical Constraint**: Serial1 is the ONLY hardware UART available. It MUST be used exclusively for GRBL communication. No Serial.println() debug output can use Serial1.

### Technical Stack
[Source: docs/architecture.md#Tech Stack Alignment]

- **Hardware**: Arduino MKR WiFi 1010 (ATSAMD21 Cortex-M0+ @ 48MHz)
- **Serial Port**: Hardware Serial1 at 115200 baud (TX=pin 14, RX=pin 13)
- **Language**: C++ (Arduino framework, C++11 compatible)
- **Memory Constraint**: 32KB SRAM total - minimize dynamic allocation
- **GRBL Protocol**: GRBL 1.1f specification (wait-for-ok flow control)

### Key Interfaces
[Source: docs/architecture.md#Component 1: SerialManager]

```cpp
class SerialManager {
public:
  bool begin(unsigned long baud);              // Initialize Serial1
  bool sendLine(const char* line);             // Send command, wait for ok/error
  bool waitForOk(unsigned int timeout_ms);     // Block until GRBL responds
  String getLastResponse();                     // Retrieve last error/response
  GRBLStatus parseStatus(String response);      // Parse response type
  bool isReady();                               // Check GRBL ready without blocking
  
private:
  String _lastResponse;                         // Store last GRBL response
};

enum GRBLStatus {
  GRBL_OK,
  GRBL_ERROR,
  GRBL_ALARM,
  GRBL_TIMEOUT,
  GRBL_UNKNOWN
};
```

### Coding Standards - Serial Communication
[Source: docs/architecture.md#Coding Standards and Conventions]

**Non-blocking I/O:**
- Never block indefinitely waiting for Serial data
- All serial operations MUST have timeout (recommend 1000ms for GRBL responses)
- Always check return values from Serial reads/writes
- Use `readStringUntil('\n')` for GRBL line-based protocol

**Timeout Pattern:**
```cpp
bool SerialManager::waitForOk(unsigned int timeout_ms) {
  unsigned long startTime = millis();
  while (millis() - startTime < timeout_ms) {
    if (Serial1.available()) {
      String response = Serial1.readStringUntil('\n');
      if (response.startsWith("ok")) {
        return true;
      } else if (response.startsWith("error")) {
        _lastResponse = response;
        return false;
      }
    }
  }
  return false; // Timeout
}
```

**Response Parsing:**
- GRBL responses are line-based (terminated with '\n')
- "ok" indicates command accepted
- "error:X" indicates error (X is error code 1-9)
- "ALARM:X" indicates alarm condition (requires $X to clear)
- Status reports: `<Idle|MPos:0.000,0.000,0.000|...>` (handle separately if needed)

### GRBL Protocol Reference
[Source: docs/prd.md#Compatibility Requirements CR1]

**GRBL 1.1f Protocol Compatibility**:
- Baud rate: 115200
- Flow control: Wait-for-ok (send command, wait for "ok" response before next command)
- Response format: Line-based, \n terminated
- Error codes: error:1 through error:9 (see GRBL docs for meanings)
- Alarm codes: ALARM:1 through ALARM:9 (hard/soft limits, probe fail, etc.)
- Settings query: Send "$$" to get all settings (27 settings, $0-$26 plus axis settings $100-$131)
- Settings modify: Send "$X=value" to change setting X

### Risk Mitigation
[Source: docs/prd.md#Risk Assessment RISK 5]

**GRBL Response Parsing Robustness**: 
- Current implementation uses `startsWith("ok")` substring matching
- Risk: False positives if unexpected messages contain "ok" substring
- Mitigation for this story:
  - Test with actual GRBL 1.1f device responses
  - Consider exact line matching if issues found
  - Handle status reports separately from command responses
